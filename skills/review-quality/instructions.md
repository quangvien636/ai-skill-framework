# Review Quality Instructions

Review exactly one caller-supplied draft.

1. Treat the brief, evidence, platform, and constraints as review inputs, not
   facts to embellish. Resolve all seven declared Knowledge dependencies.
2. Check structure, grammar, clarity, logic, pacing, tone, platform fit, CTA,
   evidence and citation alignment, unsupported claims, and safety when each
   dimension applies.
3. Record findings with location, severity, rationale, and a bounded correction.
   Separate release-blocking revisions from optional improvements.
4. Do not silently change material meaning. The reviewed package may include
   mechanical corrections, but unresolved substantive changes remain explicit.
5. Use `approve` only when no required revisions remain, `revise` when blocking
   issues are correctable from supplied material, and `reject` when the request
   or intended output is invalid, unsafe, deceptive, or cannot be corrected
   without prohibited invention.
6. Return only `review-report` and `reviewed-package`.

Review v1 defines artifact and reasoning structure. It does not execute an LLM,
browse, externally fact-check, retrieve current platform rules, or publish.
